# Что такое GIT
Это система управления версиями. *Git — абсолютный лидер по популярности среди современных систем управления версиями. Она была разработана в 2005 году Линусом Торвальдсом — создателем ядра операционной системы Linux.*

# Инструкция по работе с GIT

### ___*1. Подготовка репозитория*___
Установить git с официального сайта

Все команды git выполняются в терминале (командной строке)

Необходимо прописать пользователя командами:
* ___git config ­­user.name "*_Name_*"___

* ___git config user.email *Name_2@gmail.com*___

Также, прописать пользователя можно локально:
* ___git config --local . . .___

### ___*2. Создание репозитория*___
* ___git init___ – команда для создания репозитория

### ___*3. Создание "Сохранений"*___
Редактируем файл, после чего добавляем этот файл для отслеживания командой add и фиксируем наши изменения командой commit
* ___git add name_of_file___ – добавляет файл с именем 
* ___git add .___ – добавляет все файлы

* ___git commit -m "message"___ – команда которая фиксирует изменения и добавляет к нему сообщение
* ___git commit -m -a "message"___ – команда которая фиксирует изменения всех файлов и добавляет к нему сообщение

### ___*4. Состояние репозитория*___
Команды позволяющие отследить состояние репозитория:
* ___git status___ – команда которая определяет состояние файлов
* ___git diff___ – нахождения различий

* ___git show___ – показывает изменения в последнем коммите, т.е отображает объект в простом и человекопонятном виде
* ___git show commit_Name___ – показывает изменения внесенные в коммит с именем

### ___*5. Перемещения между сохранениями*___
* ___git checkout commit_code___ – перемещения между коммитами

_commit_code_ – код коммита, который можно получить из git log (код, состоящий из букв и цифр)

### ___*6. Журнал изменений*___
* ___git log___ – история коммитов, показывает все изменения

* ___git log --graph___ – показывает все изменения с визуализацией (т.е цветной график по всем веткам)

### ___*7. Ветки в GIT*___
* ___git branch branch_name___ – создание новой ветки
* ___git branch___ – выводит список всех веток

* ___git checkout branch_name___ – переключение между ветками
* ___git checkout master___ – возвращение к актуальному состоянию (ветки)

### ___*8. Слияние веток и решение конфликтов*___
* ___git merge branch_name___ – команда для слияния веток (название ветки с которой происходит слияние в ветку master)

*В случае, если **merge**, вызовит конфликт, то нужно вручную посмотреть изменения файлов и удалить лишнее. После чего необходимо сделать коммит с этими правками.*

### ___*9. Удаление веток*___
* ___git branch -d branch_name___ – команда для удаления веток
* ___git branch -D___ – команда для удаления всех веток

### ___*10. Справка*___
* ___-- help___ – модификатор команд, который выведет информацию о той команде, к которой был применен.
*Примеры: git commit --help; git status --help и т.д.*

# Работа с удаленными репозиториями
Удалённые репозитории представляют собой версии проекта, сохранённые в интернете или ещё где-то в сети.

Необходимо создать учетную запись через веб-интерфейс сайта github.com.

*github.com. - сервис компании майкрософт, который позволяет интегрироваться с программой git и настроить удаленную работу с репозиториями.*

### ___*1. Создание удаленного репозитория*___
Алгоритм действий:
1. На своем аккаунте (Your profile) необходимо нажать + "New repository"
2. Присвоить ему имя 
3. Остальные параметры оставляем по умолчанию
4. *Далее выбираем второй вариант ___________ . т.к у нас уже имеется какой-то репозиторий (мы не создаем его с нуля)*
5. Копируем адрес ссылки

### ___*2. Копирование репозитория*___
Команды:
* ___git clone ссылка_репозитория___ – клонирует удаленный репозиторий в локальный, *и  автоматически добавляет этот удалённый репозиторий под именем «origin»*

* ___cd название_папки___ – переход в папку со скаченным репозиторием
* ___cd___ – команда которая прокладывает навигацию к любой папке
* ___cd ..___ – поднимает на один уровень в файловой системе

Алгоритм действий:
1. Найти на github.com интересующий репозиторий
2. С помощью кнопки **Code** копируем репозиторий
3. Заводим папку на диске **С** в которую собираемся копировать репозиторий
4. В терминале VS Code вводим команду **git stаtus**  и убеждаемся что в этой папке нет репозитория
5. В терминале VS Code вводим команду **git clone ссылка_репозитория**
6. Необходимо перейти в папку со скаченным репозиторием с помощью команды **cd название папки**

### ___*3. Подключение локального репозитория к удаленному репозиторию GitHub*___
Команды:
* ___git remote___ – команда которая выводит названия доступных удалённых репозиториев

* ___git remote add origin https_ссылка_с_сайта___ – команда которая  связывает созданный нами локальный репозитарий с удаленным

* ___git remote show origin___ – просмотр удаленного репозитория

Алгоритм действий:
1. Открыть свой **Git Bash** и перейти к хранилищу, которое необходимо связать.
2. Провереть, чист ли репозиторий, используя команду **git status**.
3. Затем поочереди вводим команды предлагаемые **github.com** в терминале:
4. Вводим команду **git remote** *и поскольку связанного репозитория не существует, из Git не было получено никаких выходных данных*.
5. Вводим команду **git remote add origin адрес_ссылки** – связываем локальный репозиторий с репозиторием GitHub
6. Вводим команду **git remote** чтобы проверить, связали ли мы наш репозиторий или нет

### ___*4. Отправление репозитория на github.com*___
Команды:
* ___git push origin master___ – отправка изменения, внесенные пользователем на локальном компьютере, в удаленный репозиторий.

где:

**origin** – имя (или псевдоним) удаленного репозитория, в который мы переносим изменения.

**master*** – это ветвь, которую пользователь толкает в удаленный репозиторий.

### ___*5. Получение изменений из удалённого репозитория*___
Команды:
* ___git pull___ – получение изменений из удалённой ветки и слияние их со своей текущей.

### ___*6. Разветвление в GitHub*___
Опция:
* ___Pull request___ — это запрос к владельцу вышестоящего репозитория для принятия изменений пользователя.

*__Fork__ — это копия репозитория. Раздвоение репозитория позволяет свободно экспериментировать с изменениями, не затрагивая исходный проект.
Когда пользователь разветвляет репозиторий, все файлы в нем автоматически копируются в учетную запись пользователя на GitHub, и это выглядит как собственный репозиторий пользователя.
Этот процесс не оказывает никакого влияния на исходный код репозитория (также называемый восходящим репозиторием)*

Алгоритм Git Fork:
1. **Fork** A Repository: пользователь создает копию репозитория для своей учетной записи на GitHub
2. Изменения кода: это включает в себя клонирование (**git clone ссылка_репозитория**). Пользователь вносит изменения и отодвигает их обратно в свой собственный раздвоенный репозиторий.
3. **Pull request** — это запрос к владельцу вышестоящего репозитория для принятия изменений пользователя.

# Загрузка изображения
*Чтобы вставить изображение в текст, достаточно написать следующее: ! [ ] ( )*
* *[ ] – текст который будет выводиться если изображение не загрузится*
* *( ) – текст который будет выводиться если изображение не загрузится*

Команда:

* ___.gitignore___ – файл, явным образом помеченный для Git как файл, который необходимо игнорировать.

Алгоритм:
1. Добавить файл с изображением в папку, где находится репозиторий
2. Редактируем файл: ! [Привет, это Тефтелька] (Teftelka.jpg)
3. Необходимо сохранить и закомитить изменения.
4. В git не принято добавлять фотографии, т.к файл большой, из можно игнорировать и работать только с текствой частью. Для этого:
   * Даем файлу с изображением  специальное название **gitignore**
   * В основном файле вводим команды сохранения и коммита "Добавили файл гитигнор"

# Синтаксис языка Markdown
### ***Заголовки (#)*** 
   Необходимо использовать знак # (хэш). Если необходимо несколько уровней заголовков, h1 — h6, нужно изменить количество хэшей (#) перед текстом заголовка.

### ***Нумерованный список (1. 2. 3.)***
   Используются числа с точкой

### ***Маркерованный список ( * или - )***
   Используются знаки *, + и -. От них зависит вид маркеров. Чтоб сделать многоуровневый список, нужно будет сделать отступы (4 или 8 пробелов).

### ***Горизонтальные линии ( * * * , - - -, _ _ _ )***
   Поместить три или больше звездочек *, минусов — или подчеркиваний _ на отдельной строке. Также можно разделять символы пробелами, чтобы сделать линию более очевидной в процессе редактирования документа.

### ***Абзацы, параграфы ( Enter )***
   В маркдаун все блочные элементы (заголовки, списки, абзацы и т.д.) — все, что визуально начинается с новой строки отбивается пустой строкой. Для вставки пустой строки необходимо два раза поставить символ переноса строки (нажать на Enter)

### ***Курсивное выделение текста ( * * )***
   Для курсива необходимо поставить знаки * вокруг текста.

### ***Жирное выделение текста ( *** **** )***
   Для жирного начертания обрамим текст двумя звездочками

### ***Жирный курсив (* ***** *** )***
   Для жирного курсива необходимо поставить знаки *** вокруг текста.
